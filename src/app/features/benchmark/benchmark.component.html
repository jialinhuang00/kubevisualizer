§ 02

Process Tree Comparison

Three architectures for the same 528 API calls. The difference is what lives inside the process boundary — and how many TLS handshakes result.

bash + kubectl (subprocess)

58s 216 process spawns · 216 TLS handshakes · API server idle while procs spin up

Go binary (two-layer goroutines)

6s 1 TLS handshake total · 2–8 KB goroutine stack · 7 ns × 7 inner = 49 concurrent calls

Node.js @kubernetes/client-node

8s ~72 TCP handshakes · no HTTP/2 multiplexing · 7 ns × 22 calls = 154 peak in-flight
OS Process — PID, isolated memory, TLS per spawn
Goroutine — no PID, ~2KB stack, no new TLS
Promise — async I/O concurrent, CPU sequential
Connection pool — shared within process
§ 03

TCP Connection Analysis

netstat polled every 0.5s, filtering EKS endpoint IPs. TIME_WAIT holds for 60s after close — final accumulated count ≈ total unique TCP connections opened during the entire export run.

ESTABLISHED — connections alive right now. SYN_SENT — mid-handshake (transient, <200ms). TIME_WAIT — closed, waiting 60s expiry. Accumulates across the run.

bash TIME_WAIT = 3–6: 216 kubectl connections scatter across the full run. No burst of simultaneous closes. Individual connections enter TIME_WAIT and expire before the run ends.

Node.js TIME_WAIT = 25–72: the semaphore releases N namespaces at once. They all finish at roughly the same time, close ~N×22 connections simultaneously, and TIME_WAIT spikes.

Go TIME_WAIT = 3: HTTP/2 multiplexes all 528 requests over 1–3 TCP connections. Those 3 are 2 EKS IP endpoints + 1 occasional reconnect.

Why Node.js can't use HTTP/2: @kubernetes/client-node uses Node.js https (HTTP/1.1). The http2 module is a completely separate API — no automatic ALPN upgrade. Go's client-go negotiates HTTP/2 automatically. The result: Go opens 3 TCP connections for the entire export. Node.js opens 72.

Why primitives matter less than TLS here:

The export is 99% I/O wait. CPU barely registers. Adding more threads vs goroutines is a constant improvement. Switching from HTTP/1.1 (72 handshakes) to HTTP/2 (3 handshakes) is a multiplicative improvement.

Goroutines at 2KB each vs threads at 1MB each — 500× less memory. But for 200 goroutines that's 400KB vs 200MB. Significant at scale; not the reason Go wins here.

Go wins because client-go negotiates HTTP/2 via ALPN. Every other advantage is secondary.
§ 05

Token Bucket Rate Limiter

Default QPS=5, Burst=10 turned an 11s export into 103s. Why does client-go have this at all — and why don't other language clients?

Kubernetes is written in Go. client-go is not a third-party SDK — it is Kubernetes's own internal client. The same code runs inside kube-controller-manager, kube-scheduler, and every Kubernetes controller.

The rate limiter exists for controllers, not for tools. A Kubernetes controller running 200 goroutines would flood the API server if every LIST/WATCH fired unrestricted. QPS=5, Burst=10 makes sense when one controller shares an API server with 50 others. It makes no sense for a bulk export tool running alone.

Other language clients don't have this. Python's kubernetes, Java's kubernetes-client, Node.js's @kubernetes/client-node — all generated from the OpenAPI spec. They implement the HTTP surface only. They never had access to the controller internals.

The bug and the fix: clientcmd.BuildConfigFromFlags returns controller defaults baked in.
cfg.QPS = 100; cfg.Burst = 200
103s → 11s. Then namespaces=7 → 6s.

§ 06

FAQ

Go (ns=8) runs in 6s. Node.js (jobs=7) runs in 8s. Same concurrency — why the 2s gap?

Both send HTTP requests to the Kubernetes API concurrently. I/O wait is the same. The gap is what happens after the response arrives.

Go — 154 goroutines running in parallel
goroutine 1parse → write → done
goroutine 2parse → write → done
goroutine 3parse → write → done
...parse → write → done

Each goroutine owns its full pipeline. No waiting for others.

Node.js — 154 callbacks sharing one event loop
responses 1234… 154 queued
event loop [1][2][3][4]→ … → [154]

154 responses land at once. Each callback must wait its turn on the single event loop thread. That serialization is the 2s.

Do my CPU and memory affect these numbers?

Almost none. The bottleneck is network I/O, not local compute.

per request send <1ms wait API Server + network 10–70ms parse <1ms

CPU work is under 1% of total time. A machine 10× faster produces nearly the same result.
