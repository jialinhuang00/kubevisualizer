§ 03

TCP Connection Analysis

netstat polled every 0.5s, filtering EKS endpoint IPs. TIME_WAIT holds for 60s after close — final accumulated count ≈ total unique TCP connections opened during the entire export run.

ESTABLISHED — connections alive right now. SYN_SENT — mid-handshake (transient, <200ms). TIME_WAIT — closed, waiting for 60s expiry. Accumulates across the run.
Seeing high SYN_SENT during polling means a burst of new connections was opening simultaneously.

bash TIME_WAIT = 3–6: 216 kubectl connections scatter across the full run. No burst of simultaneous closes. Individual connections enter TIME_WAIT and expire before the run ends — nothing piles up.

Node.js TIME_WAIT = 25–72: the semaphore releases N namespaces at once. They all finish at roughly the same time, close ~N×22 connections simultaneously, and TIME_WAIT spikes. Higher jobs = larger spike. This is why workers=4 (23) beats node-single-j7 (72) on connection efficiency — even though single-thread is faster overall.

Go TIME_WAIT = 3: HTTP/2 multiplexes all 528 requests over 1–3 TCP connections. Those 3 are 2 EKS IP endpoints + 1 occasional reconnect. TIME_WAIT only appears when those long-lived connections finally close.

Why Node.js can't use HTTP/2: @kubernetes/client-node uses Node.js https (HTTP/1.1). The http2 module is a completely separate API — there is no automatic upgrade, no ALPN negotiation. Go's client-go negotiates HTTP/2 automatically during TLS via ALPN. The result: Go opens 3 TCP connections for the entire export. Node.js opens 72.

Why primitives matter less than TLS here:

The export is 99% I/O wait. CPU barely registers. Adding more threads vs goroutines is a constant improvement. Switching from HTTP/1.1 (72 handshakes) to HTTP/2 (3 handshakes) is a multiplicative improvement.

Goroutines at 2KB each vs threads at 1MB each — 500× less memory. But for 200 goroutines that's 400KB vs 200MB. Significant at scale; not the reason Go wins here.

Go wins because client-go negotiates HTTP/2 via ALPN. Every other advantage is secondary.

§ 05

Token Bucket Rate Limiter

Default QPS=5, Burst=10 turned an 11s export into 103s. Why does client-go have this at all — and why don't other language clients?

Kubernetes is written in Go. client-go is not a third-party SDK — it is Kubernetes's own internal client. The same code runs inside kube-controller-manager, kube-scheduler, and every Kubernetes controller. It was published as a library so you can write your own controllers using the same primitives the control plane uses.

The rate limiter exists for controllers, not for tools. A Kubernetes controller running 200 goroutines watching resources would flood the API server if every LIST/WATCH fired unrestricted. The token bucket enforces a global budget across all goroutines sharing one client. QPS=5, Burst=10 makes sense when one controller shares an API server with 50 other controllers. It makes no sense for a bulk export tool running alone.

Other language clients don't have this. Python's kubernetes, Java's kubernetes-client, Node.js's @kubernetes/client-node — all generated from the OpenAPI spec. They implement the HTTP API surface only. They never had access to the controller internals. Rate limiting falls to the API server (429 Too Many Requests) or the user.

The bug and the fix: clientcmd.BuildConfigFromFlags returns a config with controller defaults baked in. Two lines fix it:
cfg.QPS = 100; cfg.Burst = 200
103s → 11s. Then further tuning (namespaces=7) → 6s.
