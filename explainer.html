TCP Handshake Count — Measured via netstat TIME_WAIT

Each TCP connection that closes leaves a TIME_WAIT state for ~60s. Counting accumulated TIME_WAITs after an export reveals the true number of unique connections opened.
Monitor command: netstat -an | grep -E 'EKS_IP_1|EKS_IP_2' | awk '{print $6}' | sort | uniq -c

Node.js procs=4 43 Same N=4 as workers. Each OS process has a completely separate TCP stack — 11 more unique connections than worker threads.

Bash sequential / batch 0 observed Run takes 60–108s. TIME_WAIT (60s) expires mid-run. Peak ~21 ESTABLISHED per batch.
GNU parallel –jobs 8 0 observed Peak 19 ESTABLISHED, 6 SYN_SENT. 216 new TLS across 43s run — connections cycle too fast to accumulate in polling.
Surprising: workers=4 has fewer connections (32) than single-thread (84). Each worker handles a fixed set of namespaces and keeps its pool alive across all of them. Single-thread work-stealing rotates namespaces faster — connections from closing namespaces overlap with new ones opening, causing more TCP cycling. Workers win on connection efficiency, but single-thread wins on speed via work-stealing (no idle slots).

Bash has 0 accumulated TIME_WAIT — not because of reuse, but because of expiry. Bash sequential takes ~108s total. Each namespace takes ~4.5s. TIME_WAIT lasts 60s. By the time namespace 14 finishes, namespace 1's TIME_WAIT has already expired. No accumulation.

Why Node.js can't use HTTP/2: @kubernetes/client-node uses Node.js https module (HTTP/1.1). The http2 module is a completely separate API — no automatic upgrade. ALPN negotiation (which Go's client-go uses) cannot happen through the https module. Result: connections torn down between ns batches, TIME_WAIT accumulates to 84.
